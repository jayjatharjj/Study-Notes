# Java Interview Question Bank — Fundamentals (Modules 01–04)

> **How to use:** Read each question, answer it *aloud* (or write it out) first — only then read the answer. The struggle to recall is what builds interview recall. Re-run the deck until you can hit the *why* and the *gotcha*, not just the *what*. Difficulty is tagged `(easy)` / `(medium)`. Deeper treatment is linked per module.

## Language Basics & JVM

**Q1. What is Java and what does WORA mean?** *(easy)*
Java is a high-level, object-oriented, platform-independent language following "Write Once, Run Anywhere." Source compiles to bytecode (`.class`), not OS-specific machine code, so the same bytecode runs on any OS that has a JVM. The *why*: portability comes from a per-OS JVM, not per-OS recompilation.

**Q2. Why is Java platform-independent but the JVM is not?** *(medium)*
`javac` produces OS-neutral bytecode; each OS ships its own JVM that translates that bytecode to native instructions. So the *program* is portable, the *JVM* is platform-specific — Java is portable precisely *because* the JVM is not. See [module 01](../../01-java-language-basics.md).

**Q3. JDK vs JRE vs JVM?** *(easy)*
JVM executes bytecode and manages memory/GC. JRE = JVM + core libraries (enough to *run* apps). JDK = JRE + dev tools (`javac`, `jdb`, `jar`) — needed to *develop*. Relationship: `JDK ⊃ JRE ⊃ JVM`. Gotcha: you only need the JRE to run, not the full JDK.

**Q4. What is bytecode?** *(easy)*
The intermediate, platform-independent instruction set stored in `.class` files, generated by `javac`. The OS can't run it directly — the JVM interprets or JIT-compiles it to native code. It's the layer that makes WORA possible.

**Q5. What is the JIT compiler and why does it matter in microservices?** *(medium)*
Just-In-Time compiler converts "hot" (frequently executed) bytecode into native machine code at runtime and caches it, so execution speeds up over time. This is why Spring Boot apps "warm up" — the first few requests are slower than later ones under load.

**Q6. What are the JVM memory areas?** *(medium)*
Heap (objects/arrays, GC-managed, shared across threads); Stack (local variables, primitives, method frames — per-thread, auto-cleaned on return); Method Area/Metaspace (class metadata, static variables); PC Register (current instruction per thread). Gotcha: statics live in Metaspace, not the heap.

**Q7. Heap vs Stack?** *(medium)*
Heap is shared across all threads and holds objects; it's GC-managed. Stack is thread-specific, holds local variables and call frames, and is auto-cleaned when a method returns (frame popped). Each thread has its own stack; the heap is shared — that's why object state needs synchronization but local variables never do.

**Q8. What is the `main` method signature and why each part?** *(easy)*
`public static void main(String[] args)`. `public` so the JVM can call it from outside; `static` so no object is needed; `void` because it returns nothing to the JVM; `String[] args` for command-line arguments. Drop `static` and you get a runtime error at launch.

**Q9. Walk through the Java execution flow.** *(easy)*
Write `.java` → `javac` compiles to `.class` bytecode → JVM class loader loads it → static blocks run once at class init → JVM creates the `main()` stack frame → interpreter/JIT turns bytecode into native code → OS executes. See [module 02](../../02-oop-fundamentals.md).

**Q10. What does the class loader do?** *(medium)*
It loads `.class` files into the JVM's Method Area, verifies the bytecode, and prepares static fields. Classes load lazily (on first use), which is when static blocks fire. Gotcha: `NoClassDefFoundError`/`ClassNotFoundException` arise here, not at compile time.

**Q11. Why is Java a popular choice for microservices?** *(easy)*
Mature ecosystem (Spring Boot, Quarkus), JIT-backed JVM performance, a strong concurrency model, vast libraries, static typing for large codebases, and excellent tooling (IntelliJ, Maven, Gradle). The *why* interviewers want: typing + ecosystem + tooling reduce risk at scale.

## Primitives, Operators & Casting

**Q12. How many primitive types does Java have? Name them.** *(easy)*
Eight: `byte`, `short`, `int`, `long`, `float`, `double`, `char`, `boolean`. `int` is the default integer type, `double` the default floating-point. Everything else is a reference type.

**Q13. What is the size and range gotcha for `byte`?** *(easy)*
`byte` is 1 byte, range −128 to 127. Assigning 128 to a `byte` overflows/won't compile as a literal. The 8-bit signed range (−2⁷ to 2⁷−1) is a classic recall question.

**Q14. Why is `char` 2 bytes in Java when it's 1 byte in C/C++?** *(medium)*
Java `char` is a 16-bit Unicode code unit (0 to 65,535), not an ASCII byte — so Java handles international text natively. Gotcha: a single `char` cannot hold code points above U+FFFF (those need a `String`/surrogate pair).

**Q15. What's the default value of an uninitialized `int` field vs an `int` local variable?** *(medium)*
An `int` *instance/static field* defaults to `0`. An `int` *local variable* has **no** default — using it before assignment is a compile error. Gotcha: defaults apply only to fields, never to locals.

**Q16. What does integer division produce?** *(easy)*
Integer division truncates toward zero: `7 / 2 == 3`, `a / b` with ints drops the fraction. To get a decimal, make one operand floating-point: `7 / 2.0 == 3.5`. Classic off-by-precision bug.

**Q17. Difference between pre-increment and post-increment?** *(easy)*
`++x` increments then yields the new value; `x++` yields the old value then increments. In `int y = x++;` y gets the old x; in `int y = ++x;` y gets the new x. The expression *value* differs even though x ends the same.

**Q18. What are short-circuit operators?** *(medium)*
`&&` and `||` skip evaluating the right operand when the result is already determined (`false && x`, `true || x`). This is both a performance optimization and a null-guard idiom: `if (obj != null && obj.isValid())`. Their non-short-circuit cousins `&`/`|` always evaluate both sides.

**Q19. What is the ternary operator?** *(easy)*
`condition ? a : b` — a compact if/else that *returns a value*: `int max = (a > b) ? a : b;`. Unlike `if`, it's an expression you can assign or pass as an argument.

**Q20. What do `<<`, `>>` and `>>>` do?** *(medium)*
`<<` left-shifts bits (multiply by 2ⁿ); `>>` arithmetic right-shifts (divide by 2ⁿ, sign-preserving); `>>>` is the unsigned/logical right shift, filling with zeros regardless of sign. The `>>>` distinction matters for negative numbers.

**Q21. What is implicit (widening) casting?** *(easy)*
Smaller-to-larger conversion done automatically with no data loss: `byte → short → int → long → float → double`. E.g. `double d = anInt;` needs no cast. It's safe because the wider type can hold every value of the narrower one.

**Q22. What is explicit (narrowing) casting and its danger?** *(medium)*
Larger-to-smaller conversion requiring `(type)` and risking data loss: `int a = (int) 10.9;` gives `10`. Gotcha: narrowing **truncates, it does not round** — `(int) 10.9` is 10, not 11. Overflow can also silently wrap values.

**Q23. What happens with `0.1 + 0.2 == 0.3` in Java?** *(medium)*
It's `false`. `float`/`double` are IEEE-754 binary approximations, so `0.1 + 0.2` is `0.30000000000000004`. Gotcha: never use `==` on floating-point or `double` for money — use `BigDecimal` for currency.

**Q24. Where are primitives stored?** *(easy)*
Primitive *local variables* live on the stack. A primitive that's a *field* lives inside its object on the heap. Primitives store the actual value, not a reference — that's the core difference from reference types.

## Strings & String Pool

**Q25. What is a String and why is it special?** *(easy)*
An immutable sequence of characters, implemented as a `final` class. Immutability + the String pool make it safe to share, cache, and use as a map key. See [module 01](../../01-java-language-basics.md).

**Q26. What is the String Constant Pool (SCP)?** *(medium)*
A special region inside the heap that caches unique string *literals*. If `"hello"` already exists, a new `"hello"` literal reuses the same object — a memory optimization. Literals are pooled; `new String(...)` is not.

**Q27. `String s = "Java"` vs `String s = new String("Java")` — how many objects?** *(medium)*
The literal goes to (or reuses) the SCP. `new String("Java")` always creates a fresh heap object *and* references the pooled literal — so the statement involves up to 2 objects (one pooled, one on the heap). `new` deliberately bypasses pooling.

**Q28. Why is `==` dangerous for comparing Strings?** *(easy)*
`==` compares references (addresses); `.equals()` compares character content. `new String("a") == "a"` is `false` but `.equals()` is `true`. Always use `.equals()` for value comparison — the single most common Java bug at this level.

**Q29. Show the classic String identity puzzle.** *(medium)*
```java
String s1 = "Java", s2 = "Java", s3 = new String("Java");
s1 == s2          // true  — same pooled object
s1 == s3          // false — s3 is a distinct heap object
s1.equals(s3)     // true  — same value
```
Pooled literals share identity; `new` breaks it.

**Q30. Why is String immutable?** *(medium)*
Four reasons: security (safe for tokens/URLs/DB strings — can't be tampered after a check), thread safety (shareable without sync), memory optimization (SCP reuse is only safe if values can't change), and hashcode caching (computed once → fast `HashMap` keys).

**Q31. What does `s.concat(" World")` do to `s`?** *(easy)*
Nothing — it returns a *new* String and leaves `s` unchanged, because String is immutable. You must capture the result: `s = s.concat(" World");`. Forgetting this is a classic gotcha.

**Q32. How is String stored internally across Java versions?** *(medium)*
Java 8 and earlier: a `char[]`. Java 9+: "Compact Strings" — a `byte[]` plus a coder flag (ISO-8859-1 or UTF-16), saving ~30% heap for ASCII-heavy data. Behavior is unchanged; it's a memory optimization.

**Q33. What does `String.intern()` do?** *(medium)*
It returns the pooled reference for an equal string — adding the string to the pool first if absent. So `new String("hello").intern() == "hello"` is `true`. Useful to dedupe many equal strings, but overuse can bloat the pool.

**Q34. StringBuilder vs StringBuffer?** *(medium)*
Both are mutable. `StringBuffer` is synchronized (thread-safe, slower); `StringBuilder` is unsynchronized (not thread-safe, faster). Use `StringBuilder` in single-threaded code (most service methods); `StringBuffer` only when multiple threads share one builder.

**Q35. Why avoid `+=` on String inside a loop?** *(medium)*
Each `+=` creates a new String object — 1000 iterations spawn ~1000 throwaway objects, causing GC pressure. Use `StringBuilder.append()` to mutate a single object in place, then `.toString()` once at the end.

**Q36. What's the difference between `str.length()` and `arr.length`?** *(easy)*
`String.length()` is a *method* (with parentheses); array `length` is a *field* (no parentheses). Mixing them up is a frequent small slip — `arr.length()` won't compile, `str.length` won't compile.

## OOP Fundamentals (classes, objects, constructors, static)

**Q37. What is OOP and why use it?** *(easy)*
A paradigm modeling software as objects that bundle state (fields) and behavior (methods). Benefits: reusability (inheritance/composition), maintainability, scalability, and natural domain modeling. In Spring Boot, controllers/services/repositories/DTOs are OOP applied to layers.

**Q38. What are the four OOP pillars?** *(easy)*
Encapsulation (private fields + controlled access), Inheritance (`extends` for reuse), Polymorphism (overloading + overriding), Abstraction (abstract classes + interfaces). Be ready to give a one-line definition and example of each.

**Q39. Class vs Object?** *(easy)*
A class is a blueprint/template; an object is an instance built from it via `new`. `Car` defines what a car *is*; `new Car()` is an actual car on the heap. One class, many objects.

**Q40. What happens when you call `new`?** *(medium)*
JVM loads the class if needed → allocates memory on the heap → runs the constructor (after `super()` and instance initializers) to initialize fields → returns a reference stored on the stack. The object lives on the heap; the variable holds its address.

**Q41. What are the three kinds of variables?** *(medium)*
Local (inside methods, on stack, no default — must initialize), Instance (per-object, on heap inside the object, has defaults), Static (one copy per class, in Metaspace, has defaults). Belonging differs: method / object / class respectively.

**Q42. What is Java's parameter passing — by value or by reference?** *(medium)*
Always pass-by-value. For primitives it copies the value; for objects it copies the *reference*. So you can mutate an object's state via the copied reference, but reassigning the parameter (`u = new User()`) does not affect the caller. Classic trick question. See [module 02](../../02-oop-fundamentals.md).

**Q43. Demonstrate the pass-by-value gotcha.** *(medium)*
```java
void f(User u) {
    u.name = "Jay";    // visible to caller — same object
    u = new User();    // NOT visible — only the local copy reassigned
}
```
Mutation via the reference works; reassignment doesn't, because the reference itself was passed by value.

**Q44. What is a constructor and what are its rules?** *(easy)*
A special initializer with the same name as the class, no return type (not even `void`), called automatically by `new`. Its implicit first line is `super()`. It sets up the object's initial state.

**Q45. What are the types of constructors?** *(easy)*
Default (compiler-supplied when you write none), no-arg (user-defined), parameterized, and copy constructor (user-written — Java has none built in, unlike C++). All but the default are user-defined.

**Q46. When does the compiler NOT add a default constructor?** *(medium)*
As soon as you declare *any* constructor (even a parameterized one), the compiler stops providing the no-arg default. Gotcha: adding a parameterized constructor can break `new MyClass()` calls elsewhere until you add a no-arg one back.

**Q47. Can a constructor be private? Why would you do that?** *(medium)*
Yes — to prevent external instantiation. The Singleton pattern uses a private constructor plus a static `getInstance()`. Also used by factory methods and utility classes that should never be `new`'d.

**Q48. What is constructor chaining with `this()`?** *(medium)*
Calling one constructor from another in the same class via `this(args)`, which must be the first statement. It avoids duplicating init logic. You can't combine `this()` and `super()` in the same constructor since both must be first.

**Q49. Constructor vs method?** *(easy)*
Constructor: same name as class, no return type, runs automatically at `new`, initializes objects. Method: any name, must declare a return type, called explicitly, defines behavior. A "method" that looks like a constructor but has a return type is just a regular method.

**Q50. What is the `static` keyword?** *(easy)*
It makes a member belong to the class rather than any instance — one shared copy. Static variables (shared state), static methods (utility, no object needed), and static blocks (one-time class-load init). Accessed via `ClassName.member`.

**Q51. Can a static method access instance members directly? Why not?** *(medium)*
No — a static method has no `this`, so there's no instance to read instance fields/methods from. You'd have to create or receive an object first. Statics can only directly touch other static members.

**Q52. What is a static block and when does it run?** *(medium)*
A `static { ... }` block runs **once** when the class is loaded into the JVM, before `main()` and before any object is created. Used for one-time setup like loading config or initializing static state.

**Q53. What is an instance initializer block and what's the init order?** *(medium)*
A `{ ... }` block (no `static`) that runs *before the constructor body* every time an object is created. Order on `new`: `super()` → instance initializers/field inits (top to bottom) → constructor body. Useful for logic shared across multiple constructors.

**Q54. What is the `this` keyword and its three uses?** *(easy)*
A reference to the current object. Uses: (1) disambiguate field vs parameter (`this.x = x`), (2) constructor chaining (`this(...)`), (3) pass the current object as an argument (`method(this)`). It's implicit in instance methods, absent in static ones.

**Q55. Why is constructor injection preferred in Spring Boot?** *(medium)*
The dependency field can be `final` (immutable after construction), dependencies are explicit in the signature, and it's trivial to test (just pass a mock). Field `@Autowired` hides dependencies and prevents `final`. See [module 02](../../02-oop-fundamentals.md).

**Q56. What is composition (HAS-A)?** *(easy)*
One class holds a reference to another as a field and delegates work to it: `Car` HAS-A `Engine`. In Spring Boot, `OrderService` HAS-A `PaymentGateway`. It's "assembling behavior from objects," contrasted with inheritance's IS-A.

## Encapsulation & Access

**Q57. What is encapsulation?** *(easy)*
Bundling data and behavior in one class and restricting direct field access via `private` + `public` getters/setters. Callers go through a controlled API rather than touching raw state. It's the "data hiding" pillar.

**Q58. What are the four access modifiers and their visibility?** *(medium)*
`private` (same class only), default/package-private (same package), `protected` (package + subclasses), `public` (everywhere). From most to least restrictive: private → default → protected → public.

**Q59. What does "default" (package-private) access mean?** *(medium)*
No modifier written at all → visible only within the same package, *not* to subclasses in other packages. Gotcha: people assume "default = public-ish"; it's actually more restrictive than `protected`.

**Q60. Can a top-level class be `private` or `static`?** *(medium)*
No — top-level classes may only be `public` or default (package-private). `private`/`static` modifiers are allowed only on *nested* classes. Writing `private class Foo {}` at the top level is a compile error.

**Q61. Why are getters/setters better than public fields?** *(medium)*
They let you add validation, logging, lazy computation, or change the internal representation without breaking callers. A `setAge()` can reject negatives; a public field can't. Encapsulation buys you the freedom to evolve internals — and enables read-only (getter, no setter) or write-only (setter, no getter) fields.

**Q62. Encapsulation vs abstraction — what's the difference?** *(medium)*
Encapsulation *hides data* (the "how is state protected" question, via access modifiers). Abstraction *hides complexity/implementation* (the "what does it do, not how" question, via interfaces/abstract classes). One protects state; the other simplifies the contract.

**Q63. Encapsulation vs composition?** *(medium)*
Encapsulation is data hiding *within one class*. Composition is assembling behavior *across classes* (HAS-A). Encapsulation answers "how is data protected?"; composition answers "how are objects assembled?" They're orthogonal, not alternatives.

## Inheritance & Polymorphism

**Q64. What is inheritance and what does it give you?** *(easy)*
A child class acquires the non-private fields/methods of a parent via `extends`, modeling an IS-A relationship (`Dog IS-A Animal`) and enabling code reuse. The child can add or override behavior. See [module 03](../../03-oop-advanced.md).

**Q65. Which inheritance types does Java support?** *(medium)*
Single, multilevel (A→B→C), and hierarchical (one parent, many children) — all via classes. Multiple inheritance via *classes* is NOT supported (diamond problem); multiple inheritance via *interfaces* is.

**Q66. Why doesn't Java support multiple inheritance of classes?** *(medium)*
The diamond problem: if B and C both extend A and override `m()`, and D extends both B and C, the compiler can't decide which `m()` D inherits. Java sidesteps this by forbidding it for classes, allowing it for interfaces (with forced conflict resolution).

**Q67. What is NOT inherited by a subclass?** *(medium)*
Private members and constructors. The child can still trigger the parent constructor via `super()`, but it doesn't *inherit* it. Private fields exist in the object but aren't directly accessible from the child.

**Q68. What is the constructor execution order in inheritance?** *(medium)*
Parent constructor runs first, then child — because `super()` is the implicit first line of every constructor. If the parent has no no-arg constructor, the child must explicitly call `super(args)`, or it won't compile.

**Q69. What does a parent reference to a child object allow?** *(medium)*
`Parent ref = new Child();` lets `ref` call methods visible on `Parent` (inherited or overridden), running the child's overridden implementation. It cannot call child-only methods — the *reference type* decides what's visible, the *object type* decides what runs.

**Q70. What is polymorphism?** *(easy)*
The ability of one entity to take many forms. Compile-time polymorphism = method overloading (resolved by the compiler); runtime polymorphism = method overriding (resolved by the JVM via the actual object).

**Q71. What is method overloading?** *(easy)*
Same method name, different parameter lists (type, count, or order) in the same class. Resolved at compile time. Gotcha: differing only by return type is NOT valid overloading — the parameter list must differ.

**Q72. What is method overriding and its rules?** *(medium)*
A subclass redefines an inherited instance method with the same name and parameters. Rules: same/wider access modifier, same or covariant return type, can't override `static`/`final`/`private` methods. `@Override` is optional but catches mistakes.

**Q73. Overloading vs overriding — the key differences?** *(medium)*
Overloading: same class, different params, compile-time (static binding), no inheritance needed. Overriding: subclass, same signature, runtime (dynamic binding), requires inheritance. The classic one-liner: "overloading = compile-time, overriding = runtime."

**Q74. What is dynamic method dispatch?** *(medium)*
The JVM resolves an overridden instance-method call at runtime based on the actual object type, not the reference type: `Animal a = new Dog(); a.sound();` runs Dog's `sound()`. This is the mechanism behind runtime polymorphism and Spring's interface-based DI.

**Q75. What is static binding vs dynamic binding?** *(medium)*
Static (early) binding is resolved at compile time — overloaded methods, `static`, `private`, and `final` methods, and field access. Dynamic (late) binding is resolved at runtime — overridden instance methods. Statics and fields are never polymorphic.

**Q76. Can you override a static method? What happens instead?** *(medium)*
No — you *hide* it. If a child declares a static method with the same signature, the version called depends on the *reference type* (compile-time), not the object. Hence always call statics via `ClassName.method()`, never via an object reference.

**Q77. Method hiding vs method overriding?** *(medium)*
Hiding applies to static methods, resolves at compile time by reference type, and has no polymorphism. Overriding applies to instance methods, resolves at runtime by object type, and is polymorphic. `@Override` works only for overriding.

**Q78. Can a constructor be overridden?** *(medium)*
No. Constructors are not inherited, and overriding requires inheritance — so it's impossible. `super()` calls the parent constructor, but that's *chaining*, not overriding. This is a very common interview gotcha.

**Q79. Why must you override `hashCode()` whenever you override `equals()`?** *(medium)*
`HashMap`/`HashSet` find a bucket by `hashCode()` then confirm with `equals()`. If two "equal" objects return different hash codes, they land in different buckets and the map can't find duplicates — breaking lookups. The contract: equal objects must have equal hash codes.

**Q80. What does the `final` keyword do in its three forms?** *(medium)*
`final` variable = constant (can't reassign); `final` method = can't be overridden; `final` class = can't be subclassed. `String` is `final`, which is part of why it's immutable and unsubclassable. See [module 04](../../04-exception-handling-and-memory.md).

**Q81. What is the difference between `this()` and `super()`?** *(medium)*
`this()` calls another constructor in the *same* class (chaining); `super()` calls the *parent's* constructor. Both must be the first statement, so they can't appear together in one constructor.

**Q82. What is upcasting vs downcasting?** *(medium)*
Upcasting (child → parent reference) is implicit and always safe (`Animal a = new Dog();`). Downcasting (parent → child) needs an explicit cast and can throw `ClassCastException`. Guard with `instanceof` before downcasting.

**Q83. What is the `super` keyword used for?** *(easy)*
To reach the parent's members from a child: `super()` (parent constructor), `super.method()` (parent's overridden method), `super.field` (parent's hidden field). Useful when the child overrides/hides a member but still needs the parent version.

## Abstraction & Interfaces

**Q84. What is abstraction and how is it achieved?** *(easy)*
Hiding implementation details and exposing only essential behavior. Achieved via abstract classes (partial, 0–100%) and interfaces (full, 100% by design). The caller knows *what* an operation does, not *how*.

**Q85. What is an abstract class?** *(medium)*
A class declared `abstract` that can mix abstract methods (no body, subclass must implement) and concrete methods, and can hold state and a constructor. It cannot be instantiated directly. Use it for a shared base implementation plus state.

**Q86. What is an interface?** *(easy)*
A contract defining *what* a class must do, not *how*. Pre-Java 8: all methods are `public abstract`, all fields `public static final`. A class can implement many interfaces — enabling multiple inheritance of type.

**Q87. Abstract class vs interface — when to use which?** *(medium)*
Abstract class: shared state + partial implementation, single inheritance, can have a constructor. Interface: pure contract/capability, multiple inheritance, no instance state (only constants). Rule of thumb: "is-a with shared code" → abstract class; "can-do capability" → interface.

**Q88. Can interfaces have method bodies?** *(medium)*
Yes since Java 8: `default` methods (with body, inheritable/overridable) and `static` methods; Java 9+ also allows `private` methods for sharing logic among defaults. This let the JDK add methods to interfaces without breaking implementers (e.g. `List.sort`).

**Q89. What is a functional interface?** *(medium)*
An interface with exactly one abstract method, usable as a lambda target — annotate with `@FunctionalInterface` to enforce it. Examples: `Predicate<T>` (`test`), `Function<T,R>` (`apply`), `Consumer<T>` (`accept`), `Supplier<T>` (`get`). Default/static methods don't count toward the one-abstract-method rule.

**Q90. What is a marker interface?** *(medium)*
An empty interface that tags a class for special JVM/framework behavior, checked via `instanceof` at runtime. Examples: `Serializable`, `Cloneable`, `RandomAccess`. Modern code often uses annotations instead, but markers persist in the JDK.

**Q91. Can an interface have variables, and what are they implicitly?** *(easy)*
Yes — but they're implicitly `public static final` (constants). You cannot declare instance state in an interface. So `int MAX = 10;` in an interface is a public constant, not a mutable field.

**Q92. extends vs implements?** *(easy)*
A class `extends` one class and `implements` many interfaces; an interface `extends` (not implements) one or more interfaces. So multiple inheritance of *type* comes through `implements`/interface `extends`, never class `extends`.

**Q93. How do default-method conflicts get resolved?** *(medium)*
If a class implements two interfaces with the same `default` method, it must override the method explicitly (it can call a specific one via `InterfaceName.super.method()`). This forced resolution is how Java allows multiple interface inheritance safely.

**Q94. How do interfaces enable loose coupling and testing?** *(medium)*
Code depends on an interface, not a concrete class, so you can swap implementations (cached, read-only, mock) without changing callers. In Spring Boot, `UserController` depends on `UserService`; tests inject `Mockito.mock(UserService.class)`. See [module 03](../../03-oop-advanced.md).

**Q95. Why prefer composition over inheritance?** *(medium)*
Composition is more flexible (swap behavior at runtime), loosely coupled, easier to test (mock the dependency), and avoids the fragile base class problem where parent changes break children. Favor HAS-A over IS-A unless there's a true subtype relationship.

## Wrappers & Autoboxing

**Q96. What are wrapper classes and why do they exist?** *(easy)*
Object equivalents of primitives — `Integer`, `Double`, `Character`, `Boolean`, etc. They're needed where objects are required: generics/collections (`List<Integer>`), `Optional`, nullable values, and reflection. Primitives can't be used as type arguments.

**Q97. What is autoboxing and unboxing?** *(easy)*
Autoboxing: the compiler auto-converts a primitive to its wrapper (`Integer i = 5;`). Unboxing: wrapper back to primitive (`int x = i;`). It happens automatically in collections: `list.add(5); int v = list.get(0);`. See [module 03](../../03-oop-advanced.md).

**Q98. What is the Integer cache and the −128 to 127 gotcha?** *(medium)*
Java caches `Integer` objects for values −128 to 127. So `Integer a = 100, b = 100; a == b` is `true` (same cached object), but `Integer a = 200, b = 200; a == b` is `false` (distinct objects). Always use `.equals()` for wrapper value comparison — the canonical autoboxing trap.

**Q99. Why can autoboxing cause a `NullPointerException`?** *(medium)*
Unboxing a `null` wrapper throws NPE: `Integer i = null; int x = i;` blows up because the JVM calls `i.intValue()` on null. Watch for this with map lookups (`map.get(key)` returning null) and nullable DB columns.

**Q100. `int` vs `Integer` — when does the difference matter?** *(medium)*
`int` can't be null and is faster; `Integer` can be null and is required for collections/generics. Use `Integer` for nullable values (e.g. an optional DB column), `int` for guaranteed-present, performance-sensitive values. The nullability difference is the key decision factor. Avoid boxing in hot loops — each box allocates an object and adds GC pressure.

## Exceptions

**Q101. What is an exception?** *(easy)*
An event that disrupts normal program flow at runtime, represented as an object inheriting from `Throwable`. Exception handling lets a program degrade gracefully instead of crashing. See [module 04](../../04-exception-handling-and-memory.md).

**Q102. Describe the exception hierarchy.** *(medium)*
`Throwable` splits into `Error` (serious JVM/system failures — don't catch) and `Exception`. `Exception` splits into checked (compile-time) and `RuntimeException`/unchecked (runtime). `OutOfMemoryError` is an Error; `NullPointerException` is unchecked; `IOException` is checked.

**Q103. Checked vs unchecked exceptions?** *(medium)*
Checked (e.g. `IOException`, `SQLException`): the compiler forces `try-catch` or `throws`; they model recoverable external failures. Unchecked (e.g. `NullPointerException`, `IllegalArgumentException`): no mandatory handling; they model programming errors or business-rule violations. The compiler-enforcement difference is the crux.

**Q104. What's the difference between an Error and an Exception?** *(medium)*
Errors are serious, usually unrecoverable JVM/system conditions (`OutOfMemoryError`, `StackOverflowError`) you should not catch. Exceptions are application-level problems you can handle. Catching `Error` is almost always wrong.

**Q105. `throw` vs `throws`?** *(medium)*
`throw` (a statement, in the method body) actually raises an exception object: `throw new IllegalArgumentException(...)`. `throws` (in the method signature) *declares* that a method may propagate a checked exception: `void read() throws IOException`. One acts, one declares.

**Q106. How do try-catch-finally and catch ordering work?** *(medium)*
`try` holds risky code, `catch` handles by type (specific before general — the compiler enforces it), `finally` always runs for cleanup. If you put `catch (Exception e)` before `catch (IOException e)`, the specific one is unreachable and won't compile.

**Q107. Does `finally` always run?** *(medium)*
Almost always — even after `return`, `break`, or `continue` in the `try`, and even when an exception is thrown. It does NOT run on `System.exit()`, a JVM crash, or a forcibly killed thread. Treat it as always-runs for resource cleanup.

**Q108. Can `finally` override a `return` value?** *(medium)*
Yes — a `return` inside `finally` overrides a `return` in `try`/`catch`, and it also swallows any in-flight exception. This is an anti-pattern: never `return` (or throw) from `finally`, because it hides earlier results and errors.

**Q109. What is try-with-resources and why is it better than `finally`?** *(medium)*
Java 7+ syntax that auto-closes any `AutoCloseable` declared in the `try (...)` header — even on exception — closing in reverse order *before* any `finally`. Less boilerplate and no leaked resources. Use it for streams, DB connections, `Scanner`, HTTP clients.

**Q110. What is `final` vs `finally` vs `finalize()`?** *(medium)*
`final` is a keyword restricting modification (variable/method/class). `finally` is a block that always runs after try-catch. `finalize()` is a deprecated `Object` method the GC *might* call before destroying an object. They sound alike but are unrelated — a favorite trap.

**Q111. Why is `finalize()` discouraged?** *(medium)*
Its timing is unpredictable (it may never run), it can resurrect objects, it delays GC, and it's deprecated for removal (JEP 421). For cleanup, use `AutoCloseable` + try-with-resources or an explicit `close()` instead.

**Q112. When should you create a custom exception, checked or unchecked?** *(medium)*
Create one to express a domain failure clearly (`UserNotFoundException` vs a vague `RuntimeException`). Use *checked* when the caller can meaningfully recover; use *unchecked* for business-rule violations in REST services. In Spring Boot, prefer unchecked + `@ControllerAdvice`.

**Q113. How do you handle exceptions globally in Spring Boot?** *(medium)*
`@ControllerAdvice` with `@ExceptionHandler` methods centralizes exception-to-response mapping: each exception type maps to an HTTP status and a consistent `ErrorResponse` body. This removes per-controller try-catch and gives uniform error contracts. See [module 04](../../04-exception-handling-and-memory.md).

## Memory & Garbage Collection

**Q114. What is garbage collection and can you force it?** *(easy)*
Automatic reclamation of heap memory from objects no longer reachable from any GC root. You cannot force it — `System.gc()` is only a hint the JVM may ignore. The JVM decides timing. See [module 04](../../04-exception-handling-and-memory.md).

**Q115. When does an object become eligible for GC, and what are GC roots?** *(medium)*
When it's unreachable from any GC root — the trace starting points: local variables on thread stacks, static fields, active threads, JNI references. Setting `obj = null` or letting references go out of scope can make an object eligible. Reachability, not scope alone, is the rule.

**Q116. What are the GC generations and why do they exist?** *(medium)*
Young generation (Eden + Survivor) for new objects, cleaned by fast frequent Minor GC; Old/Tenured generation for long-lived objects, cleaned by slower Major/Full GC; Metaspace for class metadata. The split exploits the fact that most objects die young — so most collection is cheap.

**Q117. What is a stop-the-world pause, and which GC algorithms reduce it?** *(medium)*
A GC phase that suspends all app threads, spiking latency. G1GC (default since Java 9, balanced) limits pauses; ZGC (Java 15+) gives sub-millisecond, heap-size-independent pauses; Shenandoah is ultra-low pause; Parallel GC maximizes throughput. Tune with `-Xmx` and `-XX:MaxGCPauseMillis`.

**Q118. What replaced PermGen and why?** *(medium)*
Metaspace (Java 8+) replaced PermGen for class metadata. PermGen had a fixed size that caused frequent `OutOfMemoryError: PermGen space`; Metaspace grows in native memory by default, reducing those errors. A common "what changed in Java 8" question.

**Q119. Can Java still have memory leaks despite GC?** *(medium)*
Yes — if you keep *reachable* references to objects you no longer need (e.g. a static `Map` that grows forever, unremoved listeners, or `ThreadLocal`s not cleared). GC only collects unreachable objects; unintended reachability is the leak.

**Q120. What's the difference between `StackOverflowError` and `OutOfMemoryError`?** *(medium)*
`StackOverflowError` comes from the *stack* — usually unbounded recursion exhausting a thread's frames. `OutOfMemoryError` comes from the *heap* (or Metaspace) being exhausted. Both are `Error`s you generally fix by code/config, not by catching.

## Collections (basics)

**Q121. What is an array and why is random access O(1)?** *(easy)*
A fixed-size, same-type collection in contiguous memory. Indexed access is O(1) via address arithmetic: `address = base + index × elementSize` — a direct jump, no traversal. The trade-off: fixed size, can't grow. See [module 01](../../01-java-language-basics.md).

**Q122. Array vs ArrayList — key differences?** *(medium)*
Array is fixed-size, can hold primitives, uses `length` and `[]`. `ArrayList` is resizable, holds objects only (autoboxing for primitives), uses `size()` and `get()/add()`. Use an array for fixed-size/performance, `ArrayList` for dynamic collections.

**Q123. ArrayList vs LinkedList?** *(medium)*
`ArrayList` is backed by a resizable array: O(1) random access, slow mid-list inserts/deletes (shifting). `LinkedList` is a doubly-linked list: O(1) inserts/deletes at ends, O(n) random access. Default to `ArrayList`; `LinkedList` rarely wins in practice.

**Q124. List vs Set vs Map — the core distinction?** *(easy)*
`List` is an ordered collection allowing duplicates (indexed). `Set` disallows duplicates. `Map` stores key→value pairs with unique keys. Pick by question: ordered-with-dupes (List), uniqueness (Set), lookup-by-key (Map).

**Q125. How does HashMap work at a basic level?** *(medium)*
It stores entries in buckets indexed by the key's `hashCode()`. On `get`, it hashes the key to a bucket then uses `equals()` to find the entry. Average O(1) put/get. This is why custom keys must override both `hashCode()` and `equals()` correctly.

**Q126. What happens on a HashMap hash collision?** *(medium)*
Multiple keys land in the same bucket; entries form a linked list (and, since Java 8, convert to a balanced tree once a bucket exceeds ~8 entries), improving worst-case from O(n) to O(log n). Good hashing keeps buckets small.

**Q127. HashMap vs Hashtable?** *(medium)*
`HashMap` is unsynchronized, allows one null key and null values, and is faster. `Hashtable` is synchronized (legacy), allows no nulls. For thread safety today, prefer `ConcurrentHashMap` over `Hashtable`.

**Q128. HashSet vs TreeSet?** *(medium)*
`HashSet` is unordered, O(1) operations, backed by a `HashMap`. `TreeSet` keeps elements sorted (`Comparable`/`Comparator`), O(log n) operations, backed by a red-black tree. Use `TreeSet` only when you need ordering.

**Q129. What is a fail-fast iterator?** *(medium)*
An iterator that throws `ConcurrentModificationException` if the underlying collection is structurally modified during iteration (except via the iterator's own `remove()`). Most `java.util` collections are fail-fast. Modify via `Iterator.remove()` or iterate a copy to avoid it.

**Q130. When would you choose a `LinkedHashMap`?** *(medium)*
When you need predictable iteration order — insertion order (default) or access order (for building an LRU cache via `removeEldestEntry`). It's a `HashMap` plus a linked list across entries, costing a bit of memory for ordering guarantees.
